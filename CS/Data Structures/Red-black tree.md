# Красно-чёрное дерево

Самобалансируемое **BST** (binary search tree) с гарантированной высотой O(log n).

## Свойства

1. Каждый узел — **красный** или **чёрный**.
2. **Корень — чёрный.**
3. Все листья (NIL-сентинелы) — чёрные.
4. У красного узла **дети только чёрные** (нет двух красных подряд).
5. На каждом пути от узла до листа — **одинаковое число** чёрных узлов (black-height).

Из (4)+(5) ⇒ длиннейший путь ≤ 2 × кратчайший ⇒ высота **O(log n)**.

## Операции

| | Сложность |
|---|---|
| `search` | O(log n) |
| `insert` | O(log n) |
| `delete` | O(log n) |

`insert`/`delete` поддерживают свойства **ротациями** (left/right) и **перекрасками**. Деталей балансировки наизусть знать не надо.

## Где в JDK

- `TreeMap`, `TreeSet` — всегда красно-чёрное дерево.
- [HashMap](../../Java/Collections/HashMap.md) — бакет превращается в RB-tree при ≥ 8 коллизий и `table.length ≥ 64`. Требует `Comparable` ключи.
- `ConcurrentHashMap` — аналогично.

## Альтернативы

| Структура | Заметки |
|---|---|
| **AVL** | Строже сбалансировано, чуть быстрее `search`, медленнее `insert/delete`. |
| **B-tree / B+tree** | Дисковая память: БД (`InnoDB`, `Postgres`), файловые системы. |
| **Skip list** | Случайно сбалансированный «список со списками», O(log n) в среднем. Используется в `ConcurrentSkipListMap`. |

## Гочи

- BST → ключи должны иметь порядок (`Comparable` или `Comparator`). HashMap делает treeify только при `Comparable`.
- Не путать с **heap** (куча в смысле приоритетной очереди) — это другая структура (`PriorityQueue`).
