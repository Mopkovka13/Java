# GC Fundamentals — зачем, что собирается и как определяется мусор

## Зачем GC

Ручное управление (`malloc/free`, `new/delete`) даёт:
- **use-after-free / dangling pointer** — кто-то держит указатель на освобождённое.
- **double free** — освободил дважды.
- **memory leak** — забыл освободить.

GC меняет это на **предсказуемую цену**: паузы и оверхед, но без UB. Для бизнес-кода и больших команд — выгодный обмен.

## Reachability — кто живёт

Объект **достижим** (live), если есть путь по ссылкам **от какого-нибудь GC Root** к нему.
Недостижим → **мусор** → подлежит сбору.

```
Root ──▶ A ──▶ B ──▶ C        живые

         X ──▶ Y               
              │                
              └▶ X             цикл, но никто из Roots не указывает → оба мусор
```

## GC Roots — что это

| Категория | Пример |
|---|---|
| Локальные переменные / параметры | стек-фреймы активных методов всех потоков |
| Активные потоки | сами объекты `Thread` |
| Статические поля | через объекты `Class<?>` загруженных классов |
| JNI references | удержанные native-кодом |
| Захваченные мониторы | объект под `synchronized` |
| System ClassLoader и системные классы | внутренние корни JVM |
| String Pool | `StringTable` в HotSpot |
| JVMTI handles | debugger / profiler |

Шорткат для собеса: «**живой стек, статика, JNI, мониторы, system classes**».

## Tracing vs Reference Counting

JVM использует **tracing** (mark-sweep подход). НЕ reference counting.

| | Tracing (JVM) | Reference Counting |
|---|---|---|
| Циклы | собираются | требуют отдельного детектора |
| Стоимость каждого присваивания | 0 (write barrier — копейки) | inc/dec счётчика |
| Многопоточка | без contention на присваивании | атомарный счётчик на каждый `=` |
| Кэш-локальность | хорошая (счётчиков нет) | счётчики разбрасываются |
| Цена | пауза на сборку | размазана по всему коду |

> **Циклы в JVM — не проблема.** Это главное преимущество tracing.

## Generational Hypothesis (мост к следующей теме)

Эмпирическое наблюдение: **большинство объектов умирают молодыми**. Кто пережил несколько сборок — почти наверняка проживёт долго.

Отсюда вся архитектура HotSpot-GC: [Young/Old](Generational%20hypothesis.md), частые быстрые сборки в Young, редкие тяжёлые в Old.

## Гочи

- Циклические ссылки в JVM **собираются**. Это миф из мира Python/COM/Swift, где RC.
- «Объект, на который никто не ссылается» — упрощение. Точнее: «недостижимый из Roots».
- `null` присвоить полю — не «удалить объект». Это просто **разорвать одну ссылку**. Объект соберётся, только если станет недостижим.
- `System.gc()` — **подсказка**, не команда. JVM может проигнорировать (и часто игнорирует).
- `finalize()` — deprecated, не используется в современных GC. Заменён на `Cleaner` API.
