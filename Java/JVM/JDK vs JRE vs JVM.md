# Путь от кода до исполнения

```
.java  ──▶  javac  ──▶  .class (байткод)  ──▶  JVM  ──▶  результат
            (JDK)                              (JRE)
                                            ├─ ClassLoader
                                            ├─ Verifier
                                            ├─ Interpreter
                                            └─ JIT ──▶ нативный код
```

## 1. Пишу код → `.java`

Просто текстовый файл с исходником.

## 2. `javac` компилирует → `.class`

- `javac` — компилятор из **JDK**.
- Превращает `.java` в `.class` — **байткод** (бинарный формат для JVM).
- В реальных проектах `javac` не вызывают руками — его дёргают **Maven** (`maven-compiler-plugin`) и **Gradle** (`JavaCompile` task). Они оркестраторы: зависимости, кэш, инкрементальность — но компилятор всё тот же.
- Альтернативы: **ECJ** (Eclipse), `kotlinc`, `scalac` — тоже выдают `.class`.

## 3. `.class` — байткод

- Бинарь, не текст. Начинается с magic `0xCAFEBABE`.
- Читаемые куски внутри = имена и строки из **constant pool**.
- Посмотреть нормально:
  ```bash
  javap -c MyClass.class      # дизассемблер
  javap -v MyClass.class      # + constant pool, флаги
  ```
- Это набор команд для стековой виртуальной машины: `iconst_5`, `istore_1`, `invokevirtual` …

## 4. `java MyClass` запускает JVM

JVM = ядро. Ей нужны стандартные библиотеки → **JRE = JVM + libs**. JRE — то, что нужно, чтобы **запускать**. JDK = JRE + инструменты разработки (`javac`, `jar`, `javadoc`, `jdb`, `jshell`, `jlink`…).

> Иерархия: **JDK ⊃ JRE ⊃ JVM**.
> С Java 11 отдельный JRE Oracle не поставляет — берут JDK, под прод режут образ через `jlink`.

## 5. Внутри JVM по шагам

| Шаг | Кто делает | Что происходит |
|---|---|---|
| Загрузка | **ClassLoader** | Находит и грузит `.class` (bootstrap → platform → application) |
| Проверка | **Verifier** | Валидирует байткод (типы, стек, безопасность) |
| Подготовка | Linking | Резолвит ссылки, инициализирует static-поля |
| Исполнение | **Interpreter** | Идёт по байткоду инструкция за инструкцией |
| Ускорение | **JIT** (C1/C2 / Graal) | «Горячие» методы → нативный код процессора |
| Память | **GC** | Освобождает heap |

## 6. Две компиляции — не путать

- **`javac`**: исходник → байткод (до запуска, на сборке).
- **JIT**: байткод → нативный код (во время работы, для горячих методов).

## 7. WORA

- Байткод платформо-независим, JVM — платформо-зависима → один `.jar` идёт на Linux/Win/macOS.
- **WORA ≠ обратная совместимость** (что байткод Java 8 запустится на JVM 17/21).
- JVM зависима от платформы потому, что сама нативная: системные вызовы (epoll/WinAPI), JIT генерит код под архитектуру (x86-64/ARM64), GC работает с виртуальной памятью ОС. → Под каждую пару `ОС × архитектура` своя сборка JDK.

## 8. `jlink` — кастомный JRE

С Java 9 JDK разбит на ~70 модулей (`java.base`, `java.sql`, …). `jlink` собирает минимальный runtime только из нужных:

```bash
jdeps --print-module-deps app.jar     # узнать список модулей
jlink --add-modules java.base,java.sql,jdk.unsupported --output myruntime
```

Зачем: образ ~40–80 МБ вместо ~300 МБ JDK; меньше поверхность атаки; тонкие Docker-образы (multi-stage: jlink в первом стейдже, slim/distroless во втором).

Альтернатива на стероидах — **GraalVM Native Image** (AOT-компиляция в нативный бинарь, JVM не нужна; быстрый старт, но ограничения на рефлексию/динамику).

## Гочи

- Maven/Gradle ничего сами не компилируют — это `javac` под капотом.
- `.class` нельзя «прочитать в редакторе» — нужен `javap`.
- JIT не компилирует исходник, только байткод.
- `jlink` пакует **только runtime**, твой `.jar` лежит рядом.
