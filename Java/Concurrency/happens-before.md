# happens-before

## Что это

Формальное отношение из **JMM** между действиями (read/write/lock/unlock/start/join/…).
**A happens-before B** ⇒ всё, что сделал A, **гарантированно видно** в B.

Не «раньше по времени». Это про **видимость и упорядоченность**, которую обязан обеспечить JVM.

## Зачем нужно

Без hb-ребра JIT/CPU имеют право:
- держать значение в регистре/L1, не сбрасывать в память,
- переставлять инструкции,
- выкидывать «лишние» чтения.

Внутри одного потока это незаметно (JMM требует as-if-serial для observable результата).
**Между потоками** без hb — другой поток может никогда не увидеть запись или увидеть в произвольном порядке.

## Правила (создатели hb-рёбер)

| Правило | Edge |
|---|---|
| Program order | A перед B в коде одного потока → A hb B |
| Monitor lock | `unlock(m)` hb последующего `lock(m)` |
| Volatile | volatile write hb последующего volatile read той же переменной |
| Thread start | `t.start()` hb любого действия внутри `t` |
| Thread join | Все действия `t` hb возврата `t.join()` |
| Final fields | Корректно инициализированные `final` видны после завершения конструктора |
| Транзитивность | A hb B, B hb C ⇒ A hb C |

## Канонический пример

```java
int data = 0;
volatile boolean ready = false;

// A
data = 42;          // (1)
ready = true;       // (2) volatile write

// B
while (!ready) {}   // (3) volatile read
print(data);        // (4) → 42, гарантированно
```

Цепочка: (1) hb (2) — program order; (2) hb (3) — volatile; (3) hb (4) — program order.
Транзитивно (1) hb (4).

Убрать `volatile` → ребра (2)→(3) нет → B может вечно крутиться или увидеть `data=0`.

## Главное правило thread-safety

> Программа корректна ⇔ все **конфликтующие** доступы к общим данным упорядочены отношением happens-before.

Конфликтующие = к одному адресу, хотя бы один — запись.
Нет hb-ребра между ними → **data race** → поведение по JMM не определено.

## hb ≠ барьеры

happens-before — абстракция спецификации. Как именно её обеспечить через [[Memory barriers]] — решает компилятор. Думай в терминах hb-рёбер, не инструкций CPU.

## `synchronized` через призму hb

```java
synchronized (lock) { x = 1; }   // unlock hb …
synchronized (lock) { read(x); } // … lock → видим x=1
```

`synchronized` даёт **mutual exclusion** + **hb** между unlock и следующим lock того же монитора. Поэтому он решает и атомарность, и видимость в одном флаконе.

## Гочи

- hb **не симметрично** и **не тотально** — это частичный порядок.
- Между двумя `volatile` **разными** переменными hb-ребра нет.
- `Thread.sleep()` / `yield()` happens-before **не дают** — это не точки синхронизации.
- `final` работает только если ссылка на объект **не утекла** из конструктора до его завершения.
